關鍵洞察:repair.py 其實是一個通用的「導向式優化器」(策略庫 + 評分函式 + 候選挑選器),目前只被用在「減少可演奏性問題」。 A1/A2/A3 就是把同一個迴圈接上「難度目標」這個新目標 —— 它們是同一個 『閉環難度控制器』的三個階段。我的建議:先做 A1,緊接 C。

1 第一個動工做哪個

不管整體範圍多大,總要有一個「現在就開始」的。

A1 — 簡化 🎯 建議
enrich(加難度)的鏡像:降難度到 N 級 —— 八度收摺、 和弦瘦身成三和弦/單音、去裝飾、拉長時值。
  • 複用 repair.py 既有的 omit/octave 策略,工程量比想像小
  • 最貼專案使命 —— 老師「把譜改簡單給學生」比加難度更剛性
  • 做完就鋪好 A2/A3 的路
價值 ★★★★★   工程量 中   複用 高
C — 練習模式
熱圖找最難小節 + 段落 loop + 播放游標 + 指板模擬器 + 指法 DP → 慢速循環練最難幾小節。
  • 五個現有功能直接串,幾乎不寫新引擎
  • 一下打開「演奏者」這個新客群
價值 ★★★★☆   工程量 低   複用 極高
A2 — 難度抹平
per-measure 難度目標:太難的小節往下簡化、太簡單的往上 加厚,收斂成平滑的難度曲線。
  • 解「整首簡單卻卡一個爆難小節」的痛點
  • 依賴 A1 的簡化能力,先做它要連 A1 一起
價值 ★★★★☆   工程量 中   複用 高
B — 技能感知分譜
樂句偵測 + per-part 難度 + per-player skill_level → 難樂句分給強的演奏者,各人難度貼合級數。
  • 相當原創的編曲能力
  • 要動 arranger 的分配約束,工程量中高
價值 ★★★★☆   工程量 中高   複用 中

勾選你想在這一輪committed要做的。建議:A1 + C(投入低、開新客群)。

A1 簡化 🎯
降難度到 N 級;加難度的鏡像。
C 練習模式 🎯
慢速循環練最難小節 + 指板顯示。
A2 難度抹平
逐小節收斂到平滑難度曲線。
A3 通用控制器
閉環:全手段一起調到目標難度。
B 技能感知分譜
難樂句分給強的演奏者。
D2 品質 lint
每次編輯回報旋律/和聲/可演奏性 delta。
E 哼唱→合奏
錄音轉譜 → 改編 → 試聽,串成一條流程。

線性 undo 升級成可分支版本樹。
F MCP agent
外部 agent 跑完整自主改編對話。

3 「難度閉環」三件套(A1·A2·A3)怎麼走

A1 簡化、A2 抹平、A3 通用控制器其實是同一個「閉環難度控制器」 的三個階段。你想怎麼處理它們的關係。

一次規劃、分批實作 🎯 建議
把三者當同一個專案一起設計架構,但分階段交付 (A1 → A2 → A3)。
  • 架構一次想清楚,不會做完 A1 才發現要重構
  • 交付節奏仍可控
只先做 A1,看回饋再說
A1 上線、收使用者回饋,確認方向對了再規劃 A2/A3。
  • 範圍最小、最快驗證需求
  • A2/A3 可能要回頭微調 A1 的介面
這輪整套做完
A1+A2+A3 當成一個大功能一次出。
  • 難度控制一次到位,功能最完整
  • 這輪範圍最大、交付最慢
