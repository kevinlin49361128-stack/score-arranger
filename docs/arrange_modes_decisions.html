改編模式 — 快速 / 完整 切換

Score Arranger 方向決策 · 2026-05-29 · 基於 0.1.59 (timeout 已修, 弦四→鋼琴 35s)
背景: arrange() 本身 0.2s, 慢的全在 repair_loop (品質精修)。「快 vs 嚴謹」= 跑多少驗證+修復。repair 已參數化 (max_iterations / strategies), collect_issues 檢查可分層 (ERROR 可演奏性 / WARNING 音樂精修), 所以切換技術上乾淨。
我的建議: 2 個明確模式 + 自動預設, 不要 3 個。但「漸進式」(秒出草稿+背景精修) 可能比 toggle 更優雅 — 見 Q1。
Q1 · 互動模型 (單選) — 使用者怎麼碰到「快/慢」?
M-L A. 漸進式 — 秒出草稿, 背景精修 (無模式選擇) 🎯 我的首選

按「改編」→ 立刻 (0.2s) 顯示 raw arrange, 標「精修中…」; repair 在背景跑, 完成後 (~數秒~35s) 自動更新譜面 + 通知。使用者永遠看到最快的初稿, 又拿到最完整的結果。

為什麼是首選 / 反方
繞過「要不要選模式」的認知負擔 — 對音樂老師 / 業餘者最友善。反方: 前端要處理兩階段更新 (raw → refined), 若使用者在精修中已開始編輯會有衝突; 工程量比 toggle 大。但這是「正確的難」。
S B. 明確 toggle — 改編前選「草稿 / 完整」

改編按鈕旁加下拉: 草稿 (快) / 完整 (精修)。自動預設依譜長挑 (小譜→完整, 大譜→草稿), 可覆寫。

反方
多一個決策點; 「草稿」產出較差時使用者可能沒意識到自己選了草稿 → 抱怨「改編很爛」。需清楚標示 + 一鍵升級完整。工程量最小 (repair 已參數化)。
S C. 純自動 — 依譜長挑, 完全不給選

引擎依音符數 / 小節數自動決定 repair 深度, UI 完全不出現「模式」。小譜全力精修, 大譜自動降級保證不 timeout。

反方
最少認知負擔, 但剝奪「我就要快」或「我要最好」的控制權。進階使用者 (改編家) 會想要明確控制。
Q2 · 模式數量 (單選) — 若採 toggle (Q1=B)
2 個: 草稿 / 完整 🎯 建議

對應真實心智: 「先看能不能用」vs「印譜前精修」。2 個使用者好分辨。

3 個: 快速 / 標準 / 嚴謹

多一個「標準」中間檔。

反方
使用者分不清「標準」跟「嚴謹」差在哪 — engineering 概念漏進 UI。我不建議。

各模式跑什麼 (技術對照)

項目草稿 (快)完整 (精修)

Q3 · 優化項目 (可複選) — 讓「完整」也快
M最大結構贏 增量 collect_issues — 只重驗被改動的小節 🎯 建議

repair 內層每試一個策略就整譜重掃 (×6 策略 × 10 輪)。策略只改 1 個小節, 卻重驗全譜。改成只重驗被動到的小節 + 鄰接 → 兩個模式都受益, 甚至讓「草稿/完整」之分變得沒必要。

反方
要小心跨小節的檢查 (voice-leading 看相鄰小節邊界) — 需重驗 ±1 小節。改 repair 核心, 中風險, 要完整 regression。
S memoize 品質 LCS

quality._lcs_length (旋律相似度, O(n²)) 每輪重算。melody 線多半不變, 可快取 / 只在旋律聲部變動時重算。profile 顯示它是修完 find_region_at 後的第二瓶頸 (~1s)。

M 輕量 snapshot 取代整譜 deepcopy

每策略試錯前 copy.deepcopy(target) 整譜 (470 萬次 deepcopy 呼叫)。改成只快照被改的小節, 或記 undo-delta。

反方
deepcopy 雖多但單次便宜 (0.04s); 改 snapshot 機制風險高於收益, 優先序低於增量 collect。
S安全網 超大譜硬上限 — 永遠不再 timeout 🎯 建議

不管模式, 依音符數設 repair 時間/迭代硬上限 (e.g. >2000 音符自動降 max_iter), 並 log 告知「已自動降級」。防止任何病態譜再撞 300s。便宜的保險。

我的建議組合

務實首發: Q1=B toggle + Q2=2 個 + Q3=增量 collect + 硬上限。toggle 工程小先上, 增量 collect 把完整模式壓到 ~8s, 硬上限保證不 timeout。

理想終態: Q1=A 漸進式 + Q3 全做。秒出草稿 + 背景精修, 使用者無感最好。但工程量大, 建議分兩階段: 先 toggle, 驗證使用者真的會切, 再投資漸進式。

反方總結: 如果增量 collect 把「完整」壓到 <5s, 可能根本不需要「草稿」模式 — 直接全用完整就好, 省掉整個 toggle 的認知負擔與維護成本。先做 Q3 的增量 collect, 量完速度再決定要不要 Q1/Q2, 是最省的路徑。
